An AI agent with shell access can read your `.env` file, call any URL, and exfiltrate credentials without leaving a trace. Earl cuts those attack surfaces by giving agents a gate instead of raw shell access: they invoke named templates, secrets stay in the OS keychain, and outbound traffic follows an explicit allowlist you define.

```
┌─────────────┐      ┌──────────────┐      ┌───────────────────┐
│  AI Agent   │ ───▶ │     earl     │ ───▶ │ External Service  │
│  (Request)  │      │ (Gate/Policy)│      │ (API, DB, shell…) │
└─────────────┘      └──────────────┘      └───────────────────┘
       ▲                    │
       └──────── Result ────┘
```

## How it works

You define allowed operations as HCL templates. Each template declares the URL, method, parameters, and which secrets the operation can access. You store credentials in the OS keychain with `earl secrets set`. Your agent calls `earl call provider.command --param value`. Earl resolves secrets, checks network rules, and executes — blocking anything that doesn't match a template.

That's the whole model. Agents can do exactly what you've approved. Nothing more.

<Callout type="warn">
  Earl's guarantees hold as long as agents cannot modify config or template files.
  If an agent can edit `config.toml`, it can whitelist arbitrary network destinations.
  Lock down the config directory after setup: `chmod 700 ~/.config/earl`
</Callout>
