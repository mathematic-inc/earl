The OS keychain is the right default for individual setups. Teams running Earl on shared infrastructure usually have secrets in an existing vault — 1Password Connect, HashiCorp Vault, AWS Secrets Manager, or similar. Earl pulls from those directly using URI-scheme secret references, so you don't have to copy secrets into a local keychain just to use them in templates.

## How it works

Earl decides how to resolve a secret based on the key's format. A plain key like `github.token` goes to the OS keychain. A key starting with a URI scheme routes to the matching backend.

```hcl
# Resolved from the OS keychain
auth {
  kind   = "bearer"
  secret = "github.token"
}

# Resolved from 1Password Connect
auth {
  kind   = "bearer"
  secret = "op://work/github/token"
}

# Resolved from HashiCorp Vault (KV v2)
auth {
  kind   = "bearer"
  secret = "vault://secret/myapp#api_key"
}
```

The dispatch happens in `require_secret`: if the key contains `://`, Earl extracts the scheme and hands the full reference to the registered resolver for that scheme. If no resolver is compiled in for that scheme, Earl returns an error rather than silently falling back to the keychain.

The template author writes what secret they need (`op://work/github/token`). The operator configures the environment to provide it (`OP_CONNECT_HOST`, `OP_CONNECT_TOKEN`). Those two concerns don't need to meet at template-authoring time.

## Installation

External secret backends are compile-time features. The prebuilt release binaries do not include them — you need to build Earl from source with the features you want.

```bash
cargo install earl --features secrets-1password  # 1Password Connect
cargo install earl --features secrets-vault       # HashiCorp Vault
cargo install earl --features secrets-aws         # AWS Secrets Manager
cargo install earl --features secrets-gcp         # GCP Secret Manager
cargo install earl --features secrets-azure       # Azure Key Vault
```

You can combine multiple features:

```bash
cargo install earl --features secrets-1password,secrets-vault
```

<Callout type="info">
  The `secrets-vault` feature adds the `vaultrs` crate as a dependency. The
  `secrets-aws` feature adds the AWS SDK (`aws-config` and
  `aws-sdk-secretsmanager`). The others (`secrets-1password`, `secrets-gcp`,
  `secrets-azure`) use only dependencies already present in the tree.
</Callout>

## 1Password Connect

**URI format:** `op://vault-name/item-name/field-name` or `op://vault-name/item-name/section-name/field-name`

1Password vault and item names can contain spaces: `op://My Vault/GitHub API/token` is valid.

Earl tries authentication in this order:

1. **Connect Server** — when both `OP_CONNECT_TOKEN` and `OP_CONNECT_HOST` are set. Earl calls the Connect API directly. `OP_CONNECT_HOST` must be an `http://` or `https://` base URL with no path (e.g., `https://op.internal.example.com`).
2. **`op` CLI** — falls back to running `op read <reference>` as a subprocess. This works with `OP_SERVICE_ACCOUNT_TOKEN` (CI/CD) or an active `op signin` session (developer workstation).

```bash
export OP_CONNECT_HOST="https://op.internal.example.com"
export OP_CONNECT_TOKEN="eyJhbGciOiJFUzI1NiIsImtpZCI6..."
```

Template using a 1Password secret:

```hcl
template "github" "list_repos" {
  description = "List repos for an org"

  annotations {
    secrets = ["op://work/github/token"]
  }

  operation {
    protocol = "http"
    method   = "GET"
    url      = "https://api.github.com/orgs/{{ args.org }}/repos"

    auth {
      kind   = "bearer"
      secret = "op://work/github/token"
    }
  }
}
```

## HashiCorp Vault

**URI format:** `vault://mount/path#field`

The `mount` is the KV v2 secrets engine mount point (commonly `secret`). The `path` is the secret path within that mount. The `field` (after `#`) is the key to extract from the secret's data map. Do not include `data/` in the path — Earl's KV v2 client adds it automatically.

```
vault://secret/myapp#api_key         # mount=secret, path=myapp, field=api_key
vault://secret/team/app#db_password  # mount=secret, path=team/app, field=db_password
```

Two environment variables are required: `VAULT_ADDR` (the server address, e.g. `https://vault.example.com:8200`) and `VAULT_TOKEN` (a valid authentication token). Several optional variables are also supported:

- `VAULT_NAMESPACE` — Vault Enterprise namespace, e.g. `admin/team-a`
- `VAULT_SKIP_VERIFY` — set to `1` or `true` to skip TLS verification (not for production)
- `VAULT_CACERT`, `VAULT_CAPATH` — paths to CA certificate(s) for private CAs
- `VAULT_CLIENT_CERT`, `VAULT_CLIENT_KEY` — mTLS client certificate and key

```bash
export VAULT_ADDR="https://vault.example.com:8200"
export VAULT_TOKEN="hvs.CAESIBtR0QkD..."
```

Template using a Vault secret:

```hcl
template "stripe" "create_charge" {
  description = "Create a Stripe charge"

  annotations {
    secrets = ["vault://secret/stripe#secret_key"]
  }

  operation {
    protocol = "http"
    method   = "POST"
    url      = "https://api.stripe.com/v1/charges"

    auth {
      kind     = "api_key"
      location = "header"
      name     = "Authorization"
      secret   = "vault://secret/stripe#secret_key"
    }
  }
}
```

## AWS Secrets Manager

**URI format:** `aws://secret-name` or `aws://secret-name#json-key`

AWS secret names can include slashes: `aws://prod/db/credentials` is valid. If the secret's value is a JSON object and you only need one field, append `#json-key` to extract it. Without `#json-key`, Earl returns the raw `SecretString` value.

```
aws://prod/api-key              # returns raw secret value
aws://prod/db-creds#password    # parses secret as JSON, returns "password" field
```

Authentication uses the standard AWS SDK credential chain: `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`/`AWS_SESSION_TOKEN` environment variables, then `~/.aws/credentials`, then IAM instance profile, ECS task role, and so on. No Earl-specific configuration is needed beyond whatever your AWS environment already requires.

```bash
export AWS_REGION="us-east-1"
export AWS_ACCESS_KEY_ID="AKIA..."
export AWS_SECRET_ACCESS_KEY="..."
```

Template using an AWS secret:

```hcl
template "internal" "query_db" {
  description = "Run a read-only query"

  annotations {
    secrets = ["aws://prod/myapp-db#connection_url"]
  }

  operation {
    protocol = "sql"

    sql {
      connection_secret = "aws://prod/myapp-db#connection_url"
      query             = "SELECT id, name FROM users WHERE active = true LIMIT {{ args.limit }}"
    }
  }
}
```

## GCP Secret Manager

**URI format:** `gcp://project-id/secret-name` or `gcp://project-id/secret-name/version`

Without a version, Earl resolves `latest`, which is the highest-numbered version regardless of whether it is enabled or disabled. For production use, specify an explicit version number.

```
gcp://my-project/api-key        # latest version
gcp://my-project/api-key/3      # explicit version 3
```

Authentication uses Application Default Credentials (ADC). Earl checks, in order: `GOOGLE_APPLICATION_CREDENTIALS` (path to a service account or user credentials JSON file), then the well-known user credentials file at `~/.config/gcloud/application_default_credentials.json` (written by `gcloud auth application-default login`), then the GCE metadata server for workloads running on Google Cloud. The service account or user needs the `roles/secretmanager.secretAccessor` role on the secret.

<Callout type="warn">
  Workload Identity Federation (`external_account` credential type) is not yet
  supported. Use a service account key file or `gcloud auth
  application-default login` instead.
</Callout>

Template using a GCP secret:

```hcl
template "maps" "geocode" {
  description = "Geocode an address"

  annotations {
    secrets = ["gcp://my-project/maps-api-key"]
  }

  operation {
    protocol = "http"
    method   = "GET"
    url      = "https://maps.googleapis.com/maps/api/geocode/json"

    params {
      address = "{{ args.address }}"
      key     = "{{ secrets['gcp://my-project/maps-api-key'] }}"
    }
  }
}
```

## Azure Key Vault

**URI format:** `az://vault-name/secret-name`

Azure Key Vault names are 3–24 characters, alphanumeric and hyphens only, no consecutive hyphens, and cannot start or end with a hyphen.

Earl tries authentication in this order:

1. **Service principal** — when `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, and `AZURE_CLIENT_SECRET` are all set
2. **Managed Identity** — via App Service/Functions/Container Apps (`IDENTITY_ENDPOINT` + `IDENTITY_HEADER`), then via IMDS at `169.254.169.254` (VMs, VMSS, AKS). Set `AZURE_CLIENT_ID` without `AZURE_CLIENT_SECRET` to select a user-assigned identity.
3. **Azure CLI** — `az account get-access-token` for developer workstations with an active `az login` session

For sovereign clouds, set `AZURE_AUTHORITY_HOST` (default: `login.microsoftonline.com`) and `AZURE_VAULT_SUFFIX` (default: `vault.azure.net`).

```bash
export AZURE_TENANT_ID="12345678-..."
export AZURE_CLIENT_ID="abcdef01-..."
export AZURE_CLIENT_SECRET="..."
```

Template using an Azure Key Vault secret:

```hcl
template "azure" "send_notification" {
  description = "Send a notification via Azure Communication Services"

  annotations {
    secrets = ["az://my-vault/comms-connection-string"]
  }

  operation {
    protocol = "http"
    method   = "POST"
    url      = "https://myservice.communication.azure.com/sms"

    auth {
      kind     = "api_key"
      location = "header"
      name     = "Authorization"
      secret   = "az://my-vault/comms-connection-string"
    }
  }
}
```

## Constraints

`earl secrets set` cannot write to URI-scheme keys. External secret references are read-only from Earl's perspective — the value lives in the external backend, and Earl only reads it at request time. Attempting to set a URI key returns an error:

```
$ earl secrets set op://work/github/token
Error: secret key `op://work/github/token` looks like an external secret reference (contains ://).
External secrets are read-only and managed by their respective secret managers.
Use `earl secrets set <name>` for keychain secrets only.
```

Similarly, `earl secrets get` and `earl secrets list` show only keychain-stored keys. External references don't appear in the metadata index because Earl never stores their values locally.

## Troubleshooting

**1Password:** If both `OP_CONNECT_TOKEN` and `OP_CONNECT_HOST` are missing, Earl attempts the `op` CLI. If `op` is not on `PATH`, you'll get a clear error listing both options. If the vault or item name is ambiguous (multiple items match), Earl returns an error asking you to use the UUID instead.

**HashiCorp Vault:** The most common issue is paths that accidentally include `data/`. The Vault KV v2 API prepends `data/` automatically — if your path is `data/myapp`, it becomes `secret/data/data/myapp` at the API level. Use `vault://secret/myapp#field`, not `vault://secret/data/myapp#field`.

**AWS:** A missing or wrong `AWS_REGION` / `AWS_DEFAULT_REGION` causes `ResourceNotFoundException` even when the secret exists — it just exists in a different region. Set the region explicitly.

**GCP:** `PERMISSION_DENIED` from the Secret Manager API means the credentials don't have `roles/secretmanager.secretAccessor`. The `latest` version alias resolves to the highest-numbered version even if disabled — use an explicit version if you need a specific one.

**Azure:** If service principal authentication fails and you're not on Azure infrastructure, Earl falls back to the Azure CLI. Run `az login` and `az account set --subscription <id>` to ensure the correct account is active. For managed identity on VMs, the IMDS probe uses a 2-second timeout and fails fast if not on Azure — that's expected behavior.
