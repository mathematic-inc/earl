An agent with full shell access to the `earl` binary still cannot read your secret values. That guarantee holds because secrets never leave the OS keychain — macOS Keychain, Linux Secret Service, or Windows Credential Manager — and nothing in Earl writes them to disk in plaintext. The `earl secrets get` command returns metadata only: key name and timestamps. The actual value is always `[REDACTED]`. There is no flag or command to print it. Output templates have no access to the `secrets.*` namespace, so a template cannot echo a secret in its response even if the author tries. If a secret value somehow appears in raw API response data, Earl's redactor catches it before the output reaches you or an agent, checking the raw value plus its base64, hex, and URL-encoded forms.

The one thing you need to provide is a running keychain daemon on Linux. `gnome-keyring`, `kwallet`, or `keepassxc` with Secret Service enabled will all work. If none is running, `earl secrets set` will fail with a keyring error. See [Troubleshooting](/docs/troubleshooting) for setup guidance.

## Storing secrets

Run `earl secrets set` with a key name and Earl prompts you for the value interactively. The prompt does not echo input to the terminal.

```bash
earl secrets set github.token
```

For CI/CD pipelines or scripts where an interactive prompt is not possible, use `--stdin`:

```bash
printf '%s' "$GITHUB_TOKEN" | earl secrets set github.token --stdin
```

The `--stdin` flag reads from standard input and trims trailing newlines.

Key names use dot notation by convention — `provider.field` — so `github.token`, `stripe.api_key`, and `mydb.connection_url` are all typical. The key you pass to `earl secrets set` must exactly match the key referenced in your template.

## Managing secrets

`earl secrets list` prints a table of all stored key names with their created and updated timestamps — no values. `earl secrets get <key>` shows the same metadata for a single key; if the key does not exist, Earl exits with an error. `earl secrets delete <key>` removes the secret from both the OS keychain and the metadata index; if the key did not exist, Earl prints a message and exits normally.

Earl tracks which keys exist in a metadata index at `~/.local/state/earl/secrets-index.json`. This file contains only key names and timestamps, never values. It is written with `0600` permissions on Unix and is updated automatically on every `set` or `delete` operation.

## External secret managers

If your team stores credentials in 1Password Connect, HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager, or Azure Key Vault, Earl can pull from those directly using URI-scheme references. Instead of `github.token` you would write `op://work/github/token`, and Earl resolves the value at request time from the configured provider. See [External Secret Managers](/docs/external-secrets) for setup.

## OAuth2

OAuth2 is for APIs that issue short-lived access tokens rather than static keys. You define named profiles in `config.toml` and Earl handles login, token storage, automatic refresh, and injection into requests. Earl supports three flows — choose based on what the API requires and what your environment allows.

<Tabs>
  <Tab value="auth_code_pkce" label="Auth Code + PKCE">
    Use this flow when you are authenticating as a user and have a browser available. Earl opens a browser, the user authorizes, and a local callback server receives the code. Needs `client_id`, `token_url`, and `authorization_url`.

    ```toml
    [auth.profiles.github]
    flow = "auth_code_pkce"
    client_id = "your-github-client-id"
    token_url = "https://github.com/login/oauth/access_token"
    authorization_url = "https://github.com/login/oauth/authorize"
    scopes = ["repo", "read:org"]
    ```

    The default callback URL is `http://127.0.0.1:8976/callback`. Override it with `redirect_url` if your OAuth app is registered with a different address.

    If `device_authorization_url` is also configured and the browser-based flow fails (for example, in a headless environment), Earl automatically falls back to the device code flow.

  </Tab>
  <Tab value="device_code" label="Device Code">
    Use this flow in headless environments or when you cannot open a browser. Earl displays a URL and a user code; the user visits the URL on any device to authorize. Needs `client_id`, `token_url`, and `device_authorization_url`.

    ```toml
    [auth.profiles.azure]
    flow = "device_code"
    client_id = "your-azure-client-id"
    token_url = "https://login.microsoftonline.com/common/oauth2/v2.0/token"
    device_authorization_url = "https://login.microsoftonline.com/common/oauth2/v2.0/devicecode"
    scopes = ["User.Read"]
    ```

  </Tab>
  <Tab value="client_credentials" label="Client Credentials">
    Use this flow for machine-to-machine authentication with no user interaction. Needs `client_id`, `token_url`, and `client_secret_key` — the name of a secret already stored in the keychain.

    ```toml
    [auth.profiles.internal_api]
    flow = "client_credentials"
    client_id = "service-account-id"
    client_secret_key = "internal_api.client_secret"
    token_url = "https://auth.internal.example.com/oauth/token"
    scopes = ["api:read", "api:write"]
    ```

    Store the secret with:

    ```bash
    earl secrets set internal_api.client_secret
    ```

    Earl fetches the first token automatically when the profile is used — you do not need to run `earl auth login` for this flow. When a token expires and no refresh token is available, Earl re-authenticates without prompting.

  </Tab>
</Tabs>

### Profile configuration reference

Every profile requires `flow`, `client_id`, and `token_url` (or an `issuer` that provides it via OIDC discovery). Additional fields depend on the flow:

| Field                      | Type   | Description                                                                                                                                                                                               |
| -------------------------- | ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `flow`                     | string | **Required.** One of `auth_code_pkce`, `device_code`, or `client_credentials`.                                                                                                                            |
| `client_id`                | string | **Required.** OAuth2 client identifier.                                                                                                                                                                   |
| `token_url`                | string | Token endpoint URL. Required unless `issuer` provides it via OIDC discovery.                                                                                                                              |
| `authorization_url`        | string | Authorization endpoint. Required for `auth_code_pkce`. Can be discovered via `issuer`.                                                                                                                    |
| `device_authorization_url` | string | Device authorization endpoint. Required for `device_code`. Can be discovered via `issuer`.                                                                                                                |
| `client_secret_key`        | string | Secret key in Earl's keychain for the client secret value.                                                                                                                                                |
| `issuer`                   | string | OIDC issuer URL. Earl fetches `/.well-known/openid-configuration` to discover `token_url`, `authorization_url`, and `device_authorization_url`. Explicitly set URLs take precedence over discovered ones. |
| `redirect_url`             | string | OAuth callback URL for `auth_code_pkce`. Defaults to `http://127.0.0.1:8976/callback`.                                                                                                                    |
| `scopes`                   | list   | OAuth scopes to request. Defaults to `[]`.                                                                                                                                                                |
| `use_auth_request_body`    | bool   | Send client credentials in the request body instead of the `Authorization` header. Defaults to `false`.                                                                                                   |

### OIDC discovery

If your identity provider supports OpenID Connect, set `issuer` instead of specifying each endpoint URL by hand:

```toml
[auth.profiles.okta]
flow = "auth_code_pkce"
client_id = "your-okta-client-id"
issuer = "https://dev-123456.okta.com"
scopes = ["openid", "profile"]
```

Earl fetches `https://dev-123456.okta.com/.well-known/openid-configuration` and extracts `authorization_endpoint`, `token_endpoint`, and `device_authorization_endpoint`. Any URL you set explicitly in the profile overrides the discovered value.

The issuer URL goes through the same SSRF validation as other OAuth endpoints — it must use HTTPS (or HTTP to a loopback address).

### Auth commands

`earl auth login <profile>` starts the OAuth flow defined in the profile: opens a browser for `auth_code_pkce`, displays a verification URL for `device_code`, or exchanges credentials directly for `client_credentials`.

`earl auth status <profile>` shows whether a token is stored, when it expires, and which scopes were granted.

`earl auth refresh <profile>` forces a token refresh using the stored refresh token. For `client_credentials` flows without a refresh token, Earl re-authenticates automatically. For other flows without a refresh token, Earl prints an error directing you to run `earl auth login`.

`earl auth logout <profile>` deletes the stored token for the profile from the OS keychain.

### Token storage

OAuth tokens are stored in the OS keychain under the key `oauth2.<profile>.token` as a JSON object containing `access_token`, `refresh_token` (if the provider returned one), `token_type`, `expires_at`, and `scopes`. Tokens are treated as expired 30 seconds before their actual expiration time to account for clock skew. Debug output via `RUST_LOG` redacts `access_token` and `refresh_token` to prevent accidental credential logging.

### Endpoint security

All OAuth endpoint URLs are validated before use: they must use `https` (or `http` to loopback addresses for local development) and must resolve to public IP addresses. The same validation applies to OIDC discovery URLs. See [Security Model](/docs/security#ssrf-protection) for the full list of blocked IP ranges.

## Referencing secrets in templates

The `secrets.*` namespace is available during operation rendering — in `auth` blocks, headers, URLs, and body expressions. It is not available in `result.output`.

```hcl
annotations {
  secrets = ["github.token"]
}
operation {
  auth {
    kind   = "bearer"
    secret = "github.token"
  }
}
```

The `annotations.secrets` block tells Earl which secrets to prefetch. The `auth` block injects the value as a `Bearer` header at request time. After the request is sent, the value is discarded from memory.

<Tabs>
  <Tab value="api_key" label="API Key">
    ```hcl
    auth {
      kind     = "api_key"
      location = "header"
      name     = "X-API-Key"
      secret   = "stripe.api_key"
    }
    ```

    API keys can be placed in a `header`, `query` parameter, or `cookie`.

  </Tab>
  <Tab value="basic" label="Basic Auth">
    ```hcl
    auth {
      kind            = "basic"
      username        = "deploy"
      password_secret = "registry.password"
    }
    ```
  </Tab>
  <Tab value="oauth2" label="OAuth2 Profile">
    ```hcl
    auth {
      kind    = "oauth2_profile"
      profile = "github"
    }
    ```

    When the template runs, Earl loads the stored token, checks expiration, refreshes if needed, and injects the access token as a `Bearer` header. If no token is stored and the flow is not `client_credentials`, Earl prints an error directing the user to run `earl auth login <profile>`.

  </Tab>
  <Tab value="sql" label="SQL Connection">
    ```hcl
    operation {
      sql {
        connection_secret = "mydb.connection_url"
        query = "SELECT * FROM users LIMIT 10"
      }
    }
    ```

    For SQL templates, the database connection URL is referenced as a secret rather than set directly in the template.

  </Tab>
</Tabs>

<Callout type="warn">
  Never put secret values directly in template files. Always store them with
  `earl secrets set` and reference by key name.
</Callout>
